### RLHF Illustration
- 프롬프트 데이터셋
	- 입력 프롬프트 $x$: “A dog is...”
	- 이 프롬프트를 2가지 모델에 넣을 수 있음
- Initial LM 초기 언어 모델: 참조 모델로 사용되는 초기 LLM 체크포인트
	- SFT까지만 끝난 모델
	- 고정된 레퍼런스 모델이 됨.
- Policy model 미세 조정된 언어 모델(RL 정책): 학습 중인 정책 모델
	- 아웃풋은 리워드 모델로 들어감.
- Reward 보상(선호도) 모델: 생성된 응답에 점수를 부여
	- 리워드 모델만 중점을 두면 기존 LM의 성능을 잃을 수도 있음. 
	- 그래서 참조 모델과 비교 
- 강화학습 업데이트(예: PPO)  

$$\theta \leftarrow \theta+\nabla_{\theta}J(\theta)$$

- 참조 모델과 정책 모델의 차이를 제한하는 KL 페널티 (레퍼런스에서 너무 벗어나면 패널티 부과)  

$$-\lambda_{\mathrm{KL}}D_{\mathrm{KL}}\left(\pi_{\theta}(y\mid x),\Vert,\pi_{\mathrm{ref}}(y\mid x)\right)$$

- 보상 모델이 출력하는 응답 점수  

$$r_{\phi}(y\mid x)$$

- 보상 점수와 KL 페널티를 결합하여 정책 모델의 학습 목적함수를 구성한다.
- 그림 출처: Hugging Face RLHF 블로그

### Reward Model Setup
- 목표: 보상 모델을 학습하여 $y_w$에 $y_l$보다 더 높은 보상을 부여한다.
- 보상 모델의 출력: $r_\phi(x,y)$
- 마지막 토큰의 표현 $h_n^L$에 선형 계층을 적용하여 스칼라 보상을 학습한다.
- $L$번째 트랜스포머 블록은 입력 토큰 $w_1,w_2,\ldots,w_n$에 대한 은닉 표현 $h_1^L,h_2^L,\ldots,h_n^L$을 생성한다.
- 입력 형식: $\texttt{[USER]}\ x\ \texttt{[ASSISTANT]}\ y$

- 작은 랭귀지 모델임.
-  $y_w$: 위닝 솔루션
-  $y_l$: 루징 솔루루션


### Reward Model Training
- Bradley-Terry 쌍별 비교 목적함수:  

$$\mathcal{L}_{\mathrm{RM}}(r_\phi)=-\mathbb{E}_{(x,y_w,y_l)\sim\mathcal{D}}\left[\log\sigma\left(r_\phi(x,y_w)-r_\phi(x,y_l)\right)\right]$$

- $r_\phi(x,y_w)$: 선호 응답의 보상
- $r_\phi(x,y_l)$: 비선호 응답의 보상
- 선호 응답의 보상이 비선호 응답의 보상보다 커지도록 보상 모델을 학습한다.
- 시그모이드 함수:  

$$y=\sigma(x)$$

- 이진 분류 형태로 목적합수를 학습하는 방식이 가장 많이 사용 되고 있음.
- $\log\sigma()$ 에 위닝 응답과 루징 응답을 빼서 시그모이드를 지나 확률 스칼라 값을 리턴 할 수 있음. 


### Optimizing LLMs with the Reward Model
- 학습된 보상 모델은 인간 판단의 대리자 역할을 한다. 보상이 높을수록 인간이 더 선호하는 응답이다.
- LLM 정책 모델이 생성한 응답의 기대 보상을 최대화한다.  

$$\max_{\theta}\mathbb{E}_{y\sim p_{\theta}(\cdot\mid x)}\left[r_{\phi}(x,y)\right]$$

- $p_{\theta}(y\mid x)$: LLM 출력 확률
- $r_{\phi}(x,y)$: LLM이 생성한 응답의 보상
- 보상 모델이 불완전하다면 어떻게 될까?
	- 리워드 모델의 성능을 올리는 연구도 매우 활발하지만 불완정성을 해소하는 연구도 많이 진행.
	- 모델 안정성에 중요


### Issues with Naïve Optimization of Rewards
- 보상 모델은 여전히 실제 인간 선호도의 근사치일 뿐이다.    
    - 잡음이 있거나 불완전할 수 있다. 예를 들어, 분포 밖 데이터에 대해 잘 일반화되지 않을 수 있다.        
- 보상만을 최대화하면 여러 문제가 발생한다.    
    - 보상 모델 결함 악용: LLM이 원하는 행동을 실제로 갖추지 않고도 높은 보상을 얻는 방법을 학습하여 **보상 모델을 “해킹”할 수 있다**.     
	    - 우회 경로로 점수만 높이는 방향으로 학습이 될 수 있음.
    - 모드 붕괴: LLM이 높은 보상을 얻는 좁은 출력 분포로 수렴하여 **다양성이 부족해지고, 서로 다른 상황에 일반화하지 못할 수 있다.**        
    - 사전학습 지식 손실: 보상 모델에 대한 과도한 최적화로 인해 LLM이 초기 사전학습 모델의 바람직한 특성, 예를 들어 문법과 사실성을 잊을 수 있다.


### Regularized Reward Optimization
- 초기 SFT 체크포인트에서 지나치게 멀어지는 것을 방지하기 위해 페널티를 추가한다.

$$\max_{\theta}\mathbb{E}_{y\sim p_{\theta}(\cdot\mid x)}\left[r_{\phi}(x,y)-\beta\log\left(\frac{p_{\theta}(y\mid x)}{p_{\mathrm{SFT}}(y\mid x)}\right)\right]$$

- $r_{\phi}(x,y)$: Maximize reward 보상 최대화 항
- $\beta$: 페널티의 강도를 조절하는 하이퍼파라미터
- $\log\left(\frac{p_{\theta}(y\mid x)}{p_{\mathrm{SFT}}(y\mid x)}\right)$로그 비율 항은 정책 모델이 초기 SFT 모델에서 지나치게 벗어나는 것을 방지한다.
- $p_{\theta}(y\mid x)>p_{\mathrm{SFT}}(y\mid x)$인 경우에 페널티를 부여한다.
- 기댓값 관점에서 이 페널티는 쿨백-라이블러 발산으로 표현된다.  

$$D_{\mathrm{KL}}\left(p_{\theta}(y\mid x),\Vert,p_{\mathrm{SFT}}(y\mid x)\right)$$


### Optimization with Reinforcement Learning (RL)
- 강화학습을 사용하는 이유:
	- 지도학습 데이터가 없고 보상 모델만 존재한다.
	- 보상 모델의 안내에 따라 모델이 새로운 응답 생성 가능성을 탐색하도록 장려한다.
- 최적화: 정책 경사법
	- 기대 보상이 증가하는 방향으로 매개변수를 조정하여 정책 모델인 LLM을 최적화한다.
- REINFORCE: 가장 단순한 정책 경사법  

$$\theta\leftarrow\theta+\alpha\nabla_{\theta}\log\pi_{\theta}(a\mid s)R$$

- $\alpha$: 스텝 크기
- $\pi_{\theta}$: 정책 모델인 LLM
- $a$: 행동, 즉 응답 생성
- $s$: 상태, 즉 사용자 프롬프트와 대화 기록
- $R$: 누적 보상






### Overview: Language Agents
- 언어 에이전트: 자연어를 인터페이스로 사용하여 사용자와 상호작용하고 실제 작업을 수행하는 시스템 
	- 챗봇처럼 단순한 답변이아니라 사람의 요청사항을 수행.
- LLM은 언어 에이전트의 기반 역할을 한다.    
    - 자연어 이해: 텍스트 형태의 사용자 입력을 이해하고 해석한다.        
    - 자연어 생성: 일관되고 적절한 응답이나 행동을 생성한다.        
    - 추론: 다단계 추론, 문제 해결, 의사결정을 수행한다.        
- 예시:    
    - 가상 비서: 사용자 명령을 이해하고 알림 설정, 음악 재생, 스마트 홈 기기 제어 등의 작업을 수행한다.        
    - 코드 에이전트: 코드 조각 생성, 개선점 제안, 특정 코드의 동작 설명을 통해 개발자를 지원한다.     
    - 비즈니스 운영: “마케팅 캠페인 만들기”와 같은 상위 수준의 목표를 세부 작업으로 분해하고, 정보를 검색·종합하며, 외부 API나 도구와 상호작용하여 각 단계를 자율적으로 수행한다.


### Overview: Multimodal LLMs
- 여러 유형의 데이터(예: 텍스트, 이미지, 오디오, 비디오)를 처리하고 이해한다.    
- 더 포괄적이고 맥락이 풍부한 이해 및 생성을 수행한다.    
- 멀티모달 입력 처리(일반적):    
    - 서로 다른 유형의 입력 데이터를 받아 처리한다.        
    - 예시: 이미지 내용 이해, 음성 전사 및 해석, 비디오 콘텐츠 분석, 센서 데이터의 정보 통합        
- 멀티모달 출력 생성(상대적으로 드묾):    
    - 다양한 모달리티로 출력을 생성한다.        
    - 예시: 텍스트 설명을 기반으로 사실적인 이미지 생성, 음성을 텍스트로 변환, 사용자 설명에 따른 음악 생성


### Overview: Multimodal Architecture
- 아키텍처:    
    - 모달리티별 전용 아키텍처가 필요하다. 예: 비전, 오디오, 비디오 인코더        
    - 일반적으로 LLM이 강력한 기반 모델 역할을 한다.        
- 멀티모달 융합: 서로 다른 모달리티의 정보를 결합한다.    
    - Early fusion 초기 융합: 처리 전에 서로 다른 모달리티의 원시 입력 데이터를 결합한다.     
	    - 인풋 단계에서 융합
    - Late fusion  후기 융합: 각 모달리티를 개별적으로 처리한 뒤, 생성된 표현을 나중에 결합한다.
	    - 예측을 먼저하고 예측을 종합.
- Individual model이 LLM이라고 생각 하면 됨.


### Overview: Multimodal Datasets
- **학습 데이터셋에는 서로 다른 모달리티가 쌍을 이루는 예시가 포함**되어야 하며, 이를 통해 모델이 서로 다른 데이터 유형 간의 관계를 학습한다.


### Visual Instruction Tuning
- 목표: 시각 정보와 텍스트 정보를 모두 활용하는 작업에서 지시를 따르도록 멀티모달 LLM을 미세 조정한다.    
- LLaVA(Large Language and Vision Assistant): 시각적 지시 튜닝을 위해 사전학습된 비전 인코더(예: CLIP)와 대규모 언어 모델(예: Llama)을 결합한다.
	- 비전인코더는 복잡한 모델이라기 보다 이미지가 들어왔을때 벡터로 바꿔주는 모델임.


### LLaVA: Architecture
- 투영 행렬 $W$를 학습하여 이미지 표현 $Z_v$를 텍스트 임베딩 $H_v$로 변환한다.  
- 시각 토큰 $H_v$와 텍스트 토큰 $H_q$를 연결하여 모델의 입력으로 사용한다.


- $X_v$는 이미지, $X_q$ 랭귀지 instruction이 쌍으로 있음.
- 투영 행렬 $W$은 비전의 이미지를 $H_v$로 변환
- 이미지 벡터와 텍스트 벡터가 concat 되어 같은 토큰 스페이스에서 LLM으로 input됨.

- 출력에서 질문과 답볍 형태로 나올 수 있게 학습되고 출력 됨.

### Filtering API Calls
- 일부 API 호출은 LLM의 작업 수행에 유용하지만, 그렇지 않은 호출도 있다.
- 유용한 API 호출은 일반적으로 이후 토큰을 생성할 때의 손실을 감소시킨다.
- 언어 모델 데이터: $x_{1:n}=$ “Pittsburgh is also known as”
- API 호출 샘플링: $c_i^1=$ “Pittsburgh는 어떤 다른 이름으로 알려져 있는가?”
- API 호출 실행: $r_i^1=$ “Steel City”
- API 호출 필터링:  

$$L_i(c_i^1\rightarrow\text{Steel City})<\min\left(L_i(c_i^1\rightarrow\varepsilon),L_i(\varepsilon)\right)$$

- API 호출 결과를 사용했을 때의 손실이 API 호출을 수행하지 않았을 때보다 작으므로 해당 호출을 유지한다.
- 손실을 감소시키지 않는 API 호출은 제거한다.
- API 호출 샘플링: $c_i^2=$ “Pittsburgh는 어느 나라에 있는가?”
- API 호출 실행: $r_i^2=$ “United States”  

$$L_i(c_i^2\rightarrow\text{United States})>\min\left(L_i(c_i^2\rightarrow\varepsilon),L_i(\varepsilon)\right)$$

- 이 API 호출은 손실 감소에 도움이 되지 않으므로 제거한다.
- 아무때나 툴을 사용하는것이 아니고 도움이 될때만 사용되게 필터 기능을 추가.
- $\epsilon$: 빈 값임.
- $\min\left(L_i(c_i^1\rightarrow\varepsilon),L_i(\varepsilon)\right)$ 이부분이 포인트임.
	- 정답을 빈값으로 대체한 후 그 loss도 계산해서 비교해서 loss가 낮아진다면 학습데이터로 사용.

### Code Infilling
- 동기: 코드는 한 번의 왼쪽에서 오른쪽 방향 생성으로 완성되는 경우가 드물며, 대신 반복적으로 편집하고 개선한다.    
- LLM이 왼쪽에서 오른쪽 방향의 코드 생성과 편집, 즉 마스킹 및 인필링을 모두 수행하도록 학습해야 한다.

- synthesis: 새로운 코드를 만드는 것
- infilling: 빈 코드를 채우는 것

- 학습은 똑같고 다만, 코드에 맞게 증강한다고 생각 하면 됨.


### InCoder Training: Causal Masking
- 학습 문서에서 여러 코드 구간을 샘플링한다.    
- 해당 코드 구간을 문서 끝으로 이동하고, 원래 위치는 특수 마스크 토큰으로 표시한다.    
- LLM이 마스킹된 전체 문서를 생성하도록 학습하여, **양방향 문맥**을 조건으로 삽입할 텍스트를 생성하는 방법을 학습한다.

- 디코더로 개발했는데 어떻게 양방향으로 보고 예측 할 수 있을까??
	- 코드의 일부를 선택 하고 선택된 내용은 문서 맨뒤로 이동하고 마스킹된 전체 문서를 생성하도록 학습.
	- 마스크드 문서를 이용해서 학습하게 되면 양방향 모두를 고려해서 생성할 수 있게 됨.
	- 모델의 특성을 바꾸는 것이 아니라 데이터를 적절하게 바꿔서 코딩이 가능하게 함.
